# ЁЯЫЧ Smart Elevator Control System тАФ ER Diagram

A database design project for a Smart Elevator Control Platform. This repository contains the Entity-Relationship Diagram (ERD) representing the full data model of a multi-building intelligent elevator management system.

---

## ЁЯУМ Project Overview

This design models a real-world smart elevator control platform used across corporate towers, malls, airports, hospitals and high-rise residential complexes in India.

> User presses floor button тЖТ Request is generated тЖТ Elevator is allocated тЖТ Ride is completed тЖТ Log is recorded тЖТ Maintenance is tracked

The goal was to design a **clean, scalable, and normalized** database schema focused on:

- Multi-building Infrastructure Management
- Floor-level Request Tracking
- Elevator Dispatch and Ride Allocation
- Real-time Elevator Status Monitoring
- Maintenance History Tracking
- Ride Analytics and Usage Logging

## ЁЯФА Junction Tables

### `floor_elevator`
Resolves the **Many-to-Many** relationship between `elevator` and `floor`.

| Column | Type | Role |
|---|---|---|
| `floor_elevator_id` | INT PK | Primary Key |
| `elevator_id` | INT FK | References `elevator` |
| `floor_id` | INT FK | References `floor` |
| `is_active` | BOOLEAN | Whether this elevator currently serves this floor |
| `is_priority` | BOOLEAN | Whether this elevator has priority access to this floor |

**Why this junction table exists:**
- One elevator serves **many floors**
- One floor can be served by **many elevators**
- This M:N relationship is resolved via `floor_elevator`
- Extra attributes like `is_active` and `is_priority` make it a rich junction table

## ЁЯза Design Decisions

- **Static vs Dynamic Separation**: `elevator` stores only static config data (type, capacity, registration). Dynamic data like current status lives in `elevator_status` and ride data lives in `ride_log`. This ensures elevator history is never overwritten.

- **`elevator_status` as separate entity**: Status changes frequently in real-time. Keeping it separate from `elevator` means the config table remains clean and status can be updated independently.

- **`floor_request` vs `elevator_assignment`**: A request is a passenger pressing a button. An assignment is the system dispatching an elevator to that request. These are two different events kept in separate tables. `floor_request` tracks `requested_at` and `completed_at`. `elevator_assignment` tracks `assigned_at` and `completed_at` independently.

- **`elevator_assignment` vs `ride_log`**: Assignment is the dispatch decision. The log is the completed trip record used for analytics like daily ride counts and elevator load analysis.

- **Timestamp discipline**: Every event table carries both a start and end timestamp. `floor_request` has `requested_at` and `completed_at`. `elevator_assignment` has `assigned_at` and `completed_at`. `ride_log` has `ride_start_time` and `ride_end_time`. `maintenance_tracking` has `maintenance_start_time` and `maintenance_end_time`. This makes time-based analytics clean and consistent.

- **Maintenance history preservation**: Every maintenance event is a new row in `maintenance_tracking`. Nothing is overwritten. `next_scheduled_maintenance` lives here so the full service history is always intact.

- **`current_floor_id` as FK in `elevator_status`**: Instead of storing a raw string like "B1", the current floor is stored as a proper FK to the `floor` table тАФ ensuring data integrity and enabling proper joins.

- **`building_id` removed from `floor_request`**: It is derivable through `floor_id тЖТ floor тЖТ building_id`. Storing it directly would be redundant.
